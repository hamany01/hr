# وثيقة التصميم المعماري والتقني - وكيل الذكاء الاصطناعي المحلي (Local AI Worker)

توضح هذه الوثيقة التفاصيل الفنية الكاملة لتصميم وتنفيذ **الوكيل المحلي (Python Worker)** كجزء أساسي من منصة التوظيف الذكية (ATS). تم تصميم هذا الوكيل لخفض تكاليف الـ APIs السحابية، وضمان استمرارية الخدمة كخيار إنقاذ واحتياطي (Fallback) في حال فشل الاتصال السحابي أو نفاد الرصيد.

---

## 1. الهيكل العام ومصدر الحقيقة (Architecture & Source of Truth)

تعتمد المنصة على مبدأ أن **قاعدة البيانات (PostgreSQL/Supabase) هي المصدر الوحيد للحقيقة (Single Source of Truth)**. لا يحتفظ الوكيل المحلي بالمهام في الذاكرة الخاصة به، بل يراقب ويعالج المهام المخزنة في جدول مخصص يُدعى `jobs`.

### نموذج قائمة المهام (Database-backed Job Queue)
عندما يقوم المستخدم برفع سيرة ذاتية أو عندما يحتاج النظام لتقييم إجابات مرشح، يتم إنشاء صف جديد في جدول `jobs` مع تحديد نوع المهمة وحالتها الافتراضية كـ `pending`.

---

## 2. التصميم المقترح للجداول في قاعدة البيانات (Database Schema)

لتنفيذ هذا النظام بدقة، يجب على الشركة المنفذة إنشاء الجداول التالية في Supabase:

### أ. جدول المهام `public.jobs`
يحتوي هذا الجدول على كافة مهام المعالجة (استخراج السير الذاتية أو التقييم الاحتياطي).

```sql
create table public.jobs (
  id uuid primary key default gen_random_uuid(),
  type text not null check (type in ('cv_extraction', 'evaluation')),
  status text not null default 'pending' check (status in ('pending', 'processing', 'done', 'failed', 'retry')),
  priority int not null default 5, -- الأولية (مثلاً 10 للتقييم الفوري، 5 لاستخراج السير الذاتية)
  payload jsonb not null, -- البيانات المدخلة للمهمة (مثل رابط السيرة الذاتية أو الإجابات)
  result jsonb, -- مخرجات المعالجة المنظمة
  attempts int not null default 0, -- عدد المحاولات الحالية
  max_attempts int not null default 5, -- الحد الأقصى للمحاولات قبل الفشل النهائي
  locked_by text, -- معرف الوكيل الذي يقوم بالمعالجة حالياً
  locked_at timestamptz, -- وقت قفل المهمة لمنع التكرار
  next_retry_at timestamptz, -- وقت المحاولة القادمة في حال الفشل المؤقت
  idempotency_key text unique, -- لمنع تكرار معالجة نفس الطلب
  error_message text, -- تفاصيل الخطأ في حال الفشل
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

### ب. جدول نبضات قلب الوكيل `public.worker_heartbeats`
يستخدم لمراقبة صحة ونشاط الوكلاء المحليين المتصلين بالنظام.

```sql
create table public.worker_heartbeats (
  worker_id text primary key,
  hostname text,
  last_seen_at timestamptz not null default now(),
  status text not null default 'healthy',
  meta jsonb
);
```

## 3. آلية حجز المهام الفعالة لمنع التكرار (Claiming Logic)

عند تشغيل أكثر من وكيل محلي في نفس الوقت، يجب ضمان عدم قيام وكيلين بمعالجة نفس المهمة. يتم تحقيق ذلك باستخدام ميزة `FOR UPDATE SKIP LOCKED` في PostgreSQL:

```sql
select *
from public.jobs
where status in ('pending', 'retry')
  and (next_retry_at is null or next_retry_at <= now())
order by priority desc, created_at asc
limit 1
for update skip locked;
```

**آلية العمل:** يقوم الاستعلام بالبحث عن المهام الجاهزة للمعالجة، ويقوم بقفل الصف المحدد (`FOR UPDATE`) مع تخطي أي صفوف مقفلة بواسطة وكلاء آخرين (`SKIP LOCKED`) بشكل فوري ودون انتظار.

## 4. قنوات استقبال الأحداث والمهام (Event-driven & Safety Polling)

يعتمد الوكيل على تصميم هجين لاستقبال التنبيهات لضمان معالجة فورية للمهام واستقرار كامل:

1. **في حال تشغيل الوكيل على خادم عام (VPS):**
   * يتم تفعيل **Supabase Database Webhooks** لإرسال تنبيه فوري إلى خدمة **FastAPI** التي تعمل داخل الوكيل عند إضافة أي مهمة جديدة.
2. **في حال تشغيل الوكيل داخلياً في الشركة خلف جدار حماية (Behind NAT):**
   * يتم استخدام **Supabase Realtime** عبر العميل غير المتزامن للبايثون (`acreate_client()`) للاستماع المباشر للتعديلات عبر WebSockets.
3. **شبكة الأمان الدائمة (Safety Polling):**
   * بغض النظر عن القناة المستخدمة، يقوم الوكيل بتشغيل دورة تحقق دورية (Polling) كل 30-60 ثانية للتأكد من التقاط كافة المهام العالقة أو التي لم تُعالج بسبب انقطاع مؤقت في الاتصال.

## 5. تدفق التنفيذ الفني للمهام (Execution Flows)

### أولاً: مسار معالجة واستخراج السير الذاتية (CV Parsing Pipeline)
1. يرفع المرشح سبرته الذاتية للمنصة -> يتم حفظها في Supabase Storage -> يُنشأ Job بنوع `cv_extraction`.
2. يلتقط الوكيل المحلي المهمة ويقوم بتنزيل الملف مؤقتاً بصيغة آمنة.
3. يتم فحص نوع الملف (PDF نصي، PDF ممسوح ضوئياً، Word DOCX، أو صور).
4. يُستخرج النص باستخدام مكتبات البايثون المخصصة، ويُطبق المحرك الـ OCR (قارئ النصوص من الصور) فقط عند الحاجة.
5. يُرسل النص المستخلص إلى نموذج الذكاء الاصطناعي المحلي (مثل **Ollama / Llama 3**) مع تحديد قالب المخرجات المنظم (Structured JSON Schema).
6. يتم التحقق من صحة واكتمال كائن الـ JSON المستخرج باستخدام مكتبة **Pydantic**.
7. يُرسل الناتج لقاعدة البيانات وتُحدث حالة المهمة لـ `done`.
8. **الحذف الفوري الآمن:** يُحذف الملف المؤقت من خادم المعالجة لضمان سرية البيانات.

### ثانياً: مسار التقييم التلقائي الاحتياطي (Fallback Scoring Pipeline)
1. تحاول المنصة تقييم إجابات المرشحين باستخدام الـ APIs السحابية المتوفرة في قاعدة البيانات بالترتيب.
2. في حال فشل جميع المفاتيح السحابية (بسبب تجاوز الحد المسموح أو مشاكل الدفع والاتصال)، يتم إنشاء مهمة في جدول `jobs` من نوع `evaluation` وبأولوية عالية (`priority = 10`).
3. يلتقط الوكيل المحلي المهمة فوراً.
4. يقوم النموذج المحلي بتقييم إجابات المرشح ووضع الدرجة الفنية ونقاط القوة والضعف باللغة العربية بناءً على المعايير المحملة مسبقاً.
5. تُحفظ النتيجة في قاعدة البيانات مع وسمها بـ `processed_by = local_fallback`.

## 6. حزمة المكتبات الموصى بها (Recommended Stack)

| الفئة | الأداة / المكتبة | الغرض والسبب |
|---|---|---|
| **محرك الويب** | `FastAPI + Uvicorn` | خفيف الوزن وسريع جداً، وممتاز لاستقبال الـ Webhooks وإظهار نقاط التحقق من الصحة (Health Checks). |
| **قاعدة البيانات** | `supabase-py` | المكتبة الرسمية للتعامل مع قاعدة بيانات وملفات وميزات Supabase بشكل غير متزامن (Async). |
| **قراءة الملفات** | `PyMuPDF + pdfplumber` | لاستخراج النصوص من ملفات PDF بأعلى دقة وسرعة ممكنة. |
| **قراءة ملفات Word** | `python-docx` | لقراءة واستخراج النصوص من ملفات Word بيسر. |
| **التعرف على الصور (OCR)** | `pytesseract + pdf2image + Pillow` | لتحويل ملفات PDF الممسوحة ضوئياً والصور إلى نصوص قابلة للقراءة. |
| **الذكاء الاصطناعي المحلي**| `Ollama` | لتشغيل وإدارة النماذج المحلية مفتوحة المصدر (مثل Llama 3 أو Mistral) والحصول على مخرجات بصيغة JSON منظمة. |
| **التحقق من البيانات** | `Pydantic` | لفرض والتحقق من صحة هيكلية البيانات المستخرجة قبل إرسالها لقاعدة البيانات. |
| **إدارة الأخطاء والمحاولات** | `tenacity` | لتطبيق سياسات إعادة المحاولة التلقائية عند الفشل المؤقت (Exponential Backoff). |

## 7. الاستقرار، الأمان، وإستراتيجية النشر (Hardening & Deployment)

1. **التشغيل التلقائي:** يتم نشر البرنامج كخدمة تدار عبر `systemd` في خوادم لينكس، أو عبر حاويات `Docker Compose` لضمان إعادة التشغيل التلقائي (`Restart=always`) في حال توقف الخدمة لأي سبب.
2. **مراقبة النشاط (Heartbeat):** يقوم الوكيل بتحديث حقل `last_seen_at` في جدول `worker_heartbeats` كل 30-60 ثانية ليعلم النظام الرئيسي بأنه نشط وجاهز لاستلام العمليات.
3. **أمان البيانات والخصوصية:** 
   * يُمنع استخدام مفتاح الصلاحيات الكاملة `service_role` خارج بيئة الوكيل المؤمنة بالكامل.
   * تُحذف السير الذاتية المؤقتة فوراً بعد المعالجة لمنع تسريب بيانات المتقدمين.
   * يتم تشفير الروابط وتأمين endpoints الخاصة بالـ Webhooks عبر فحص العناوين ورأس التوثيق (Secret Header Verification).
